## Part 1: Epic Review

### What Went Well ✅

**1. Architecture Decisions Were Well-Documented**
- Each story included explicit dev notes calling out critical gotchas:
  - Story 1.3: The `"order"` SQL reserved word requires quoting in all queries
  - Story 1.4: Route registration order matters (`/api/todos/reorder` must come before `/api/todos/:id`)
- This is excellent for onboarding future developers and preventing rework
- **Pattern to Repeat:** Every story should include a "Dev Notes" section with at least one documented lesson

**2. Synchronous Better-SQLite3 Adoption**
- Team correctly chose synchronous database calls (no async/await) per ARCH spec
- Simplified transaction patterns significantly
- Dev notes explicitly warned against the async pattern trap — critical decision was clearly communicated

**3. Repository Abstraction Pattern Was Clean**
- All routes completely decoupled from storage implementation (`ITodoRepository` interface)
- Makes persistence layer swappable without route changes
- Strong foundation for future capabilities (e.g., PostgreSQL migration, caching layer)

**4. Validation Centralized at Route Level**
- Fastify JSON Schema validation + input sanitization (trim, deduplicate tags) happens before any repository call
- Zero risk of data corruption from malformed input
- Moves validation burden away from frontend

**5. Test Coverage Built In from Day One**
- Unit tests for repository layer use in-memory SQLite fixtures (fast, isolated tests)
- Integration tests for routes test full server startup and HTTP contract
- 70% coverage threshold is now a hard requirement enforced by CI

**6. Docker Multi-Stage Build Best Practices**
- Both frontend and backend Dockerfiles use multi-stage builds (build stage + runtime stage)
- Non-root user execution for security
- Health checks configured from the start (prevents debugging headaches later)

### Challenges & Learning Moments 🔄

**1. No Major Pivots Required**
- The foundation work was specification-driven and straightforward
- Minor trade-offs were expected (e.g., adding `order` and `tags` columns now for future epics vs. adding them later)
- Team consensus: future-proofing the schema now is better than thrashing the database in Epic 3

**2. DevOps Readiness**
- Elena noted that having health checks from the start will save debugging time when multiple services need coordination
- CORS environment variable approach (set via docker-compose) provides flexibility without config file changes

**3. Data Shape Design**
- Alice questioned whether adding `order`, `tags`, `createdAt`, `updatedAt` in Epic 1 was premature when Epic 2 only uses `id`, `title`, `completed`
- Charlie's insight: Better to have the schema ready now than to change it mid-Epic 3 during tags work
- Repository abstraction means frontend won't see unused fields anyway

## Part 3: Key Insights & Lessons Learned

### Insight 1: Dev Notes Are Retrospective Gold
The pattern of documenting critical gotchas in story notes (like the `"order"` quoting issue) made this retrospective richer and will help future developers. This should be a team standard.

**Lesson:** Make "Dev Notes" a required section in every story file going forward.

### Insight 2: Architecture Decisions Reduce Risk
The repository abstraction pattern, the environment variable approach to configuration, the health check from day one — these were all up-front architectural choices that paid dividends. They prevented:
- Route handlers from being tightly coupled to SQLite
- Configuration brittleness
- Later debugging friction with service startup

**Lesson:** Time spent on good architecture decisions early is time saved on rework later. The team understood this and executed well.

### Insight 3: Test Infrastructure Prevents Quality Regression
The 70% coverage threshold in Vitest, the integration tests for routes, the use of in-memory SQLite for unit tests — these create a quality baseline that will be hard to break going forward. Subsequent epics won't accidentally introduce untested code.

**Lesson:** Quality gates should be enforced by tooling (CI threshold), not just code review.
